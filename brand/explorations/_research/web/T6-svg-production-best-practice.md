# T6 — Production best practice for a font-independent SVG brand kit: distribution matrix, SVG a11y, favicon/PWA set, print/CMYK, and path-safe optimization

*Research track T6 · web survey + local Pillow verification · compiled 2026-06-07.*
*Scope: what a professional brand kit actually ships once its masters are font-independent (outlined) — the format matrix (SVG/PDF/EPS/PNG), SVG accessibility metadata, multi-size `.ico` and favicon/PWA/maskable icon set, print deliverables (PDF/X, CMYK survival of the gold `#C8A15A`, nearest coated metallic spot), and SVG optimization (svgo/picosvg) that does not destroy outlined paths. This is the **distribution layer on top of F10** — F10 (`brand/explorations/_research/fix-specs/F10-outlined-masters-integration.md`) already owns the generator, the `outline.py` outliner, `brand/dist/outlined/`, and the build/exports policy. T6 does not re-derive any of that; it specifies what the outlined masters become downstream. Two deterministic facts (gold→CMYK, `.ico` write) were verified locally with Pillow 11.3.0, not recalled. Where a current claim could not be confirmed from a primary page it is flagged.*

## Findings

### F1 — A professional kit ships a small, deliberate format **matrix**, not "all formats" — vector masters + raster fallbacks, with EPS as a request-driven extra
The durable rule is: **keep a vector master as the single source of truth and derive everything else** (Shopify, *How To Choose the Right Logo File Format*, 2025 — https://www.shopify.com/blog/logo-file-format ; Renderforest — https://www.renderforest.com/blog/logo-file-formats ). The roles split cleanly: **SVG** = web/scalable; **PNG (transparent)** = digital placements that need a raster (email, slides, social, app stores); **PDF** = sharing/approval/print delivery (stores vector *and* raster, color-profile-aware); **EPS** = the format "print shops, signage companies, and merchandise vendors will most often request" (FreeLogoServices — https://www.freelogoservices.com/blog/logo-file-types-png-svg-eps-guide/ ; nineblaess, *Logo file formats* — https://www.nineblaess.de/blog/logo-file-formats/ ). EPS is **legacy-but-persistent**: current guidance increasingly leads with PDF+SVG and treats EPS as a vendor-compatibility deliverable for signage/embroidery/older RIPs rather than a primary format (same sources). AI/PSD "editable source" formats are proprietary and not relevant to an open, generated kit — the equivalent "source" here is the spec + live-text SVG (F10), so the kit should ship **SVG (live-text + outlined), PDF, PNG, and EPS-on-request**, skipping AI/PSD.

### F2 — SVG accessibility: the authoritative pattern is `role="img"` + a non-empty accessible name, and `aria-labelledby` → `<title>`+`<desc>` is the most robust way to supply it for inline SVG
The W3C WAI ACT rule **7d6734** ("SVG element with explicit role has non-empty accessible name") is the normative test: any SVG with an explicit role of `img` / `graphics-document` / `graphics-symbol` that is in the accessibility tree **must** have a non-empty accessible name, suppliable by `<title>`, `aria-label`, **or** `aria-labelledby` (https://www.w3.org/WAI/standards-guidelines/act/rules/7d6734/ ). Two load-bearing computation facts from that rule: **SVG `<text>` content does NOT contribute to the accessible name**, and an **empty `<title>` yields an empty name** (fails the rule). Practitioner testing converges on one pattern for inline SVG with a rich description: `svg role="img"` + `<title>` + `<desc>` + `aria-labelledby` pointing at both — "the most reliable choice for different browser and screen readers tested" (UXgem, *Accessible SVGs* — https://www.uxgem.com/articles/svg-accessibility ; Deque — https://www.deque.com/blog/creating-accessible-svgs/ ; Orange a11y guidelines — https://a11y-guidelines.orange.com/en/articles/accessible-svg/ ). The WAI note: until SVG Accessibility API Mappings get wider browser support, **explicit ARIA role + ARIA labelling "significantly improves" support** over relying on `<title>`/`<desc>` alone (ACT rule, above). Critical distinction: this applies to **inline** SVG (`<svg>` in the DOM). An SVG referenced via `<img src="…svg">` is exposed through the host `<img>`'s `alt`, and the SVG's internal `role`/`title` are not read — so a kit must label **at the embed site** when used as `<img>`, and **internally** when inlined (UXgem; Orange, above).

### F3 — RTAM's current SVGs are close but technically under-spec: they use `aria-label` + `<title>`/`<desc>` but do **not** wire `aria-labelledby`
Every shipped SVG uses `role="img" aria-label="…"` plus a `<title>` and a rich `<desc>` (verified in-repo: `brand/icons/favicon.svg`, `brand/icons/rtam-rdot-icon-black.svg`). This **passes** ACT 7d6734 (the `aria-label` supplies a non-empty name). But it leaves the `<desc>` orphaned from the accessible *description* and duplicates the name string in two places (`aria-label` and `<title>`) that can drift. The more robust, single-source pattern is `aria-labelledby="t d"` referencing `<title id="t">` + `<desc id="d">`, with unique per-file ids — which both removes the duplicated string and surfaces the `<desc>` as the accessible description (F2 sources). No source claims `aria-label` is *wrong*; the upgrade is robustness + de-duplication, and it fits F10's generator (ids are derivable from the asset path).

### F4 — The modern favicon/PWA set is ~5 icons + one manifest, not 20 files; SVG is the primary, ICO/PNG are fallbacks
The canonical primary source (Evil Martians, *How to Favicon*, updated through 2026 — https://evilmartians.com/chronicles/how-to-favicon-in-2021-six-files-that-fit-most-needs ) reduces the old 20+ file pile to: **`favicon.ico` (multi-size, 16/32/48), `icon.svg` (vector, theme-aware), `apple-touch-icon.png` (180×180, opaque, ~20 px internal padding), `icon-192.png`, `icon-512.png`, and a 512×512 maskable** — declared in `manifest.webmanifest`. The HTML is exactly three `<link>`s plus the manifest:
```html
<link rel="icon" href="/favicon.ico" sizes="32x32">
<link rel="icon" href="/icon.svg" type="image/svg+xml">
<link rel="apple-touch-icon" href="/apple-touch-icon.png"><!-- 180×180 -->
<link rel="manifest" href="/manifest.webmanifest">
```
The `sizes="32x32"` on the ICO line is **load-bearing**: it stops Chrome from downloading *both* the ICO and the SVG (Evil Martians; corroborated by browserux, *Favicons in 2025* — https://browserux.com/blog/guides/web-icons/favicons-best-practices.html ). SVG-as-favicon is supported by modern browsers and **can theme to dark mode** via `@media (prefers-color-scheme: dark)` inside the SVG, but is **not** universal (some Windows/Android paths) — hence the PNG/ICO fallbacks are still required (browserux; iconmaker.studio, *Favicon Best Practices 2025* — https://iconmaker.studio/blog/favicon-best-practices-2025 ). The ICO is "legacy-but-keep-one" — obsolete for rendering but still the safest universal tab fallback (browserux).

### F5 — Multi-size `.ico` generation in Python is one Pillow call (verified locally)
Verified with **Pillow 11.3.0** on this machine: a single `Image.save(path, format="ICO", sizes=[(16,16),(32,32),(48,48),...])` writes one `.ico` containing all those sub-images; reading it back via `Image.open(...).ico.sizes()` returns exactly the embedded set. If `sizes=` is **omitted**, Pillow's default embedded set is `16,24,32,48,64,128,256`. So `.ico` needs no external tool — the existing `brand/tools/` stack (Pillow is already a dependency) can emit it. Pillow API ref: https://pillow.readthedocs.io/en/stable/handbook/image-file-formats.html#ico . (Local check, 2026-06-07: `sizes=[(16,16),(32,32),(48,48)]` → readback `[(16,16),(32,32),(48,48)]`; default → `[(16,16),(24,24),(32,32),(48,48),(64,64),(128,128),(256,256)]`.) **Source matters:** all `.ico` sub-images are ≤64 px, so per the brand's own scale rule (favicon's bindu detaches above 64 px → use `favicon.svg` below 64, the R-monogram above), the `.ico` must be rasterized from **`favicon.svg`** (the small-tuned master) at each target size — **not** from a 256² render of the R-monogram (that is exactly the 8× detachment case the brand book warns against). The R-monogram is the source only for the >64 px deliverables (apple-touch-icon 180, manifest 192/512).

### F6 — Maskable icons: the safe zone is a centered circle of **80% diameter (40% radius)**; ship maskable **separately** from `any`
MDN (primary — https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps/How_to/Define_app_icons ) defines the maskable safe zone as "a circle which diameter is **80% of the icon's minimum dimension**" — i.e. keep all critical content within the central 80% (10% padding each side); for a 512 icon that is a **409×409** safe square / circle (Evil Martians, above, states the 409 figure explicitly). Maskable icons must have an **opaque background** filling the whole canvas (web.dev — https://web.dev/learn/pwa/web-app-manifest ; logofoundry — https://logofoundry.app/blog/pwa-icon-requirements-safe-areas ). **Conflict surfaced (Rule 8):** MDN/web.dev *permit* a combined `"purpose": "any maskable"`, but Chrome's installability guidance and multiple 2025 practitioner sources **discourage** combining them — a maskable icon has built-in padding, so reusing it as `any` renders the mark too small (logofoundry; web.dev's own caution). Resolution: ship **separate** files — `purpose:"any"` (full-bleed) and `purpose:"maskable"` (padded) — which satisfies both the spec and the stricter guidance. Recommended minimum manifest set: **192 (any), 512 (any), 512 (maskable)** (MDN; web.dev). RTAM's framed circle icons (`rtam-rdot-icon-circle-*.svg`) are already padded and are the natural maskable base.

### F7 — The gold `#C8A15A` **survives** CMYK as a flat tan/ochre, but its *metallic sheen* does not — that requires a spot metallic ink or foil
Two axes with opposite answers:
- **Flat color (verified locally):** `#C8A15A` = RGB(200,161,90) is a desaturated tan/ochre well **inside** CMYK gamut. Naive (no-ICC-profile) Pillow 11.3.0 conversion = **C22 M37 Y65 K0** (raw 0–255: 55/94/165/0; round-trips back to exactly (200,161,90) — zero clipping). This is the *naive* number; **profile-accurate** values under US Web Coated SWOP v2 / FOGRA39 differ (typically adding some K and trimming Y) and must be taken from the print profile, **not** from this naive figure. So: in print, the gold reproduces fine *as a color* — no out-of-gamut surprise.
- **Metallic sheen (cannot be CMYK):** "About 30% of all Pantone spot colors cannot be reproduced accurately with standard CMYK, and **metallics are at the top of that list**" — gold spot ink "contains tiny metallic particles that catch and reflect light"; CMYK is only a flat simulation (LionDTF, *PMS Color for Gold* — https://liondtf.com/blogs/news/pms-color-for-gold ; KMS Litho, *Pantone spots & metallics* — https://www.kmslitho.co.uk/pantone-spots-and-metallics/ ). True shimmer needs a **metallic Pantone spot ink** or **foil stamping** on **coated** stock (uncoated kills the sheen) (KMS Litho; PrintNinja, *Metallic Ink Setup* — https://printninja.com/printing-resource-center/printninja-file-setup-checklist/specialty-options-setup-guides/metallic-ink-and-spot-color-setup-guide/ ). A commonly cited standard metallic gold is **PMS 871 C** (LionDTF) — but **PMS matching is proprietary, substrate- and profile-dependent; do not treat any web-quoted Pantone number as authoritative.** The correct spot must be confirmed against a **physical Pantone Metallics guide** / the printer's house match. This connects directly to two facts already in the repo: `colors.json` already designates **bronze `#9B6A2F`** as "heavier weight than gold for **print on ivory**," and the June-2026 audit found gold-on-ivory contrast at **2.18:1** (fails WCAG) — so in flat print the gold is **decorative/accent only**, never a text or load-bearing color, exactly as it already is on screen.

### F8 — Print delivery standard: PDF/X-1a is the safe, widely-requested default; PDF/X-4 if spot/transparency/ICC matters
**PDF/X-1a** forces CMYK + spot only (no RGB), embeds all fonts, **flattens transparency**, and defines trim/bleed — "the most widely requested standard… plays nicely with legacy systems and modern presses alike" (Hybrid Helix, *PDF/X-1a vs PDF/X-4 in 2025* — https://blog.hybridhelix.com/pdf-x-1a-vs-pdf-x-4-whats-the-difference-and-which-should-you-use-in-2025/ ; IMG.LY — https://img.ly/blog/what-does-print-ready-pdf-mean-understanding-pdf-x-standards-for-professional-printing/ ). **PDF/X-4** keeps live transparency, layers, ICC-managed color, and RGB/Lab/spot — better for modern prepress, packaging, and **device-independent color via ICC profiles** (same sources). For a logo that is flat fills + one spot gold and embeds outlined paths (so no font dependency — F10), **PDF/X-1a is the correct default deliverable**; offer **PDF/X-4** when the printer wants the metallic spot kept live / ICC-managed. Note both standards carry spot colors, so the metallic-gold separation (F7) survives in either.

### F9 — SVG optimization must be **path-safe**: svgo can silently alter outlined geometry; picosvg drops `<filter>`/`<mask>` and mis-handles some nested-transform clips
Because F10's distribution masters are **outlined paths**, the optimizer's path-touching plugins are the risk, not the metadata ones:
- **svgo** `convertPathData` has `floatPrecision` default **3**; dropping below 3 can visibly distort curves and **must** be eyeballed at intended size before accepting (SVGO docs — https://svgo.dev/docs/plugins/convertPathData/ ; vectosolve, *SVG Optimization 2026* — https://vectosolve.com/blog/svg-optimization-web-performance-2025 ). `mergePaths` "might create issues if two paths overlap" and changes the DOM node structure (SVGO docs — https://svgo.dev/docs/plugins/mergePaths/ ) — for the RTAM bindu (a separate `<circle>`/path that must stay independently recolorable per F10 + usage-rules §7), merging it into the letterform path would **break the bindu-recolor contract**. svgo's `preset-default` also strips `<title>`/`<desc>` and ARIA-relevant attributes unless told not to — for a brand kit these are **required** (F2), so `removeTitle`, `removeDesc`, and id/aria handling must be disabled (DevToys, *What SVGO Strips* — https://devtoys.pro/blog/svg-optimizer-guide ; SVGO preset-default — https://svgo.dev/docs/preset-default/ ).
- **picosvg** (Google Fonts' simplifier — https://github.com/googlefonts/picosvg ) flattens transforms, materializes `<use>`, and converts strokes/`fill-rule` to plain nonzero paths — useful for *maximally portable* path-only output, **but it does not support `<filter>` or `<mask>`** and has known issues applying clips under nested transforms (picosvg README + issue #200 — https://github.com/googlefonts/picosvg/issues/200 ). RTAM's SVGs use none of those features, so picosvg is *safe here*, but it is a one-way flattener (loses the live-text/grouping structure) and should run only on the **outlined distribution copy**, never the editable master. Net rule: **optimize the outlined masters with a pinned, conservative config; keep precision ≥ the render gate tolerance; never merge the bindu; never strip a11y metadata; gate every optimized file through F10's existing render-fidelity diff (max-blob ≤ 150) so any path damage fails the build.**

## Implications for RTAM

- **The kit is already font-independent-ready (F10) but ships no distribution matrix yet.** Today the repo commits only live-text SVGs (which need a browser + Google Fonts to render correctly — the core architecture caveat in repo memory). US-13/US-15 must layer F1's matrix on top of F10's `dist/outlined/`: for each primary mark, ship **outlined SVG + PDF (PDF/X-1a) + transparent PNG set + EPS-on-request**. This is the concrete thing that makes the kit usable by a print shop or a vendor who lacks Cinzel/Tiro — the exact gap repo memory flags.

- **RTAM's a11y is a pass-but-upgrade (F2/F3).** The current `role="img" aria-label` + `<title>`/`<desc>` clears the W3C ACT bar, so this is not a defect — but wiring `aria-labelledby` to `<title>`+`<desc>` (unique ids) removes the duplicated name string and surfaces the rich `<desc>` as a real description. Because F10 generates every SVG from `brand.json`, the ids and labelledby wiring are a **generator change, applied once**, not 21 hand-edits.

- **The favicon caveat already in the brand book is exactly F4/F6.** The brand book + usage-rules already say the favicon's bindu "detaches above 64 px" and to switch to the full R-monogram. That is the same problem maskable padding solves: the kit should ship a **maskable PWA icon built from the already-padded framed circle mark** (`rtam-rdot-icon-circle-*.svg`), keeping the bindu inside the 80% safe zone, and a full-bleed `any` icon from the plain R-monogram — two files, per F6, never one combined `any maskable`.

- **In flat print the gold behaves exactly as it already does on screen — decorative only (F7).** The naive CMYK (C22 M37 Y65 K0) and the 2.18:1 ivory contrast failure already in the audit point the same way: gold is an accent, never text or a load-bearing color, in print as on screen. For sacred/foil surfaces the brand book *already* invokes embossing/foil — F7 gives that a concrete spec: a **metallic spot (candidate PMS 871 C, to be confirmed against a physical guide) or foil on coated stock**, with **bronze `#9B6A2F`** (already in the palette as the print-on-ivory alternate) as the flat-CMYK stand-in when no metallic/foil is available. This needs no new color — it operationalizes two tokens the palette already defines.

- **Phase-2 "ṛta wheel" cosmogram raises every T6 cost at once.** A multi-element cosmogram (shivling/lotus/12-medallion) is harder to keep path-safe through svgo, harder to fit the maskable 80% safe zone, harder to reduce to a legible 16/32/48 `.ico`, and — if it leans on gradients/metallic to read — *cannot* survive flat CMYK or 1-bit print. T6's deliverables (favicon-scale `.ico`, maskable safe-zone, flat-CMYK proof) should be **mandatory pre-scoring proofs** for any cosmogram candidate, reinforcing T1's reproduction-floor criterion rather than duplicating it.

## Implied repo changes (MANDATORY)

Below, **(gen)** = generator-side, lands in F10's `brand.json`/templates/`build.sh` (US-13); **(dist)** = a new publishing/distribution stage downstream of `build.sh` that consumes the outlined masters — wherever that story is numbered.*

1. **(dist) usage-rules.md — add §10 "Distribution formats" (asset spec / matrix).** Add: *"Every primary mark ships in a fixed matrix derived from the outlined master (F10 `dist/outlined/`): **outlined SVG** (font-independent, for web + vendors), **PDF/X-1a** (print delivery, CMYK + spot, transparency flattened), **transparent PNG** at 256/512/1024, and **EPS on request** for signage/embroidery/legacy RIPs. Never hand-author a derivative — all are regenerated by `build.sh`. Editable 'source' = the live-text SVG + `brand.json`, not AI/PSD."* (Basis: F1, F8.)

2. **usage-rules.md §7 (The bindu) — add an optimization-safety line.** Append: *"The bindu's independence is also an SVG-optimization constraint: never run a path-**merge** pass (svgo `mergePaths`, or any tool that unions shapes) that fuses the bindu `<circle>` into the letterform path — it must stay a separately addressable, separately recolorable node. Optimization keeps float precision ≥ the render-gate tolerance and must never strip `<title>`/`<desc>`/aria."* (Basis: F9; ties to existing §7 recolor rule.)

3. **usage-rules.md — add §11 "Accessibility metadata."** Add: *"Inline SVG marks carry `role="img"` + a non-empty accessible name. Canonical pattern: `aria-labelledby="<id-title> <id-desc>"` referencing `<title>` and `<desc>` with file-unique ids (W3C ACT rule 7d6734). When a mark is embedded via `<img src>`, label it with the host `<img alt>` instead — the SVG's internal title is not exposed there. Never ship an empty `<title>`."* (Basis: F2, F3.)

4. **Deliverables list / asset spec — add the favicon + PWA icon set.** Spec: *"Ship the modern 5-icon set: `favicon.ico` (multi-size 16/32/48, generated by Pillow `Image.save(format='ICO', sizes=[…])` rasterized from **`favicon.svg`** — all sub-images are ≤64 px so they use the small-tuned favicon master, not the R-monogram, per the brand's bindu-detachment-above-64px rule), `icon.svg` (theme-aware via `prefers-color-scheme`), `apple-touch-icon.png` (180×180, **opaque** ivory ground, ~20 px internal padding, from the R-monogram), and a `manifest.webmanifest` with **separate** `192 any`, `512 any`, and `512 maskable` PNGs (all >64 px → from the R-monogram / framed circle mark). Maskable = built from the framed circle mark with all ink inside the central 80% (409×409 of 512) safe zone; opaque ground. Never combine `any maskable` in one file."* HTML link block per F4. (Basis: F4, F5, F6.)

5. **Deliverables list / print spec — add the gold print rule.** Spec: *"Flat-CMYK gold `#C8A15A` = naive C22 M37 Y65 K0 (no profile; take profile-accurate values from US Web Coated SWOP v2 / FOGRA39, not this figure) — in gamut, reproduces as a tan/ochre but **loses all metallic sheen**. For sacred/foil surfaces specify a **metallic Pantone spot (candidate PMS 871 C — confirm against a physical Metallics guide / printer house match) or foil on coated stock**. Flat fallback where no metallic/foil is available = **bronze `#9B6A2F`** (already the palette's print-on-ivory alternate). Gold remains **accent-only** in print (2.18:1 ivory contrast fails WCAG — never text)."* (Basis: F7.)

6. **rubric.md — add criterion "C-Production / Distribution Readiness" (create `brand/explorations/_rubric/rubric.md` if T1 hasn't; do not duplicate T1's C-Reproduction Floor — cross-reference it).** Scoring anchors:
   - **5** — candidate produces a complete clean distribution set: path-safe outlined SVG (passes render gate ≤150), a legible 16/32/48 `.ico`, a maskable variant fitting the 80% safe zone, and a flat-CMYK/foil print spec with a named spot — all from one master.
   - **3** — produces the set but needs per-format hand-fixing (e.g. an extra padded variant for maskable, or precision tuning to stay path-safe).
   - **1** — cannot be reduced to a font-independent path master, fails maskable/favicon scale, or relies on gradient/metallic that collapses in flat CMYK / 1-bit print.
   (Basis: F1, F6, F7, F9. Cross-ref T1 C-Reproduction Floor for the monochrome/small-scale axis.)

7. **Risk log — add three risks.**
   - *"svgo/picosvg path damage: an over-aggressive optimization pass (low `floatPrecision`, `mergePaths`, evenodd→nonzero) can silently distort outlined glyphs or fuse the bindu. Mitigation: pinned conservative config + gate every optimized file through F10's render-fidelity diff (max-blob ≤150) so path damage fails the build."* (F9.)
